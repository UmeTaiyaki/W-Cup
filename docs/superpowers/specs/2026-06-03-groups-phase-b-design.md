# Phase B 設計：グループ順位予想・ノックアウト予想・新採点

- 日付: 2026-06-03
- 対象: W杯2026 予想アプリ（Cloudflare Pages `wcup2026-yosou`）
- 前提: Phase A（48カ国12グループ・admin・読み取り専用グループタブ）は実装/デプロイ済み
- 関連: `wcup-bracket-2026.md`（ベスト32トーナメント構造）, `2026-06-03-groups-phase-a-design.md`

## 1. 目的とスコープ

「優勝・準優勝・得点王」のコア予想に加え、**やりたい人だけ**が取り組めるオプション予想（グループ順位／ノックアウト勝ち上がり）を追加する。コア体験は一切変えない。

### スコープ内（ユーザー側）
- グループ順位予想UI（12組・順番タップ式）
- 3位ワイルドカード割当UI（8枠・手動）
- ノックアウト予想UI（R32→決勝・ハイブリッド：モバイル=ステッパー／デスクトップ=フルブラケット）
- 新採点ロジック（コア部門／総合部門の2部門）
- ランキング画面の2部門タブ化
- 「予想」タブのハブ化

### スコープ外（別フェーズ）
- admin 側の「ノックアウト実結果」入力UI（採点の答え合わせに必要だが、本specの後続タスクとして扱う。グループ実順位入力は Phase A 実装済み）
- 3位決定戦（3位/4位の試合）の採点 → **対象外で確定**

## 2. 決定事項（ユーザー合意済み・2026-06-03）

| 論点 | 決定 |
|---|---|
| 3位枠の扱い | **完全手動割当**。各組1〜3位を予想 → 8枠に許可グループの3位を手動セット。各3位は1回のみ使用。FIFA対照表は実装しない |
| 順位予想UI | **順番タップ式**。4チームをタップした順に1位/2位/3位が付く。4位は残り1チームが自動 |
| 予想構造 | コア（優勝/準優勝/得点王）は独立した手動ピックで現状維持。オプションは ①順位→②3位割当→③ノックアウト の1本トラック |
| ノックアウトの対戦カード | ユーザー自身のグループ順位予想＋3位割当から自動で組む |
| ランキング | **2部門タブ**：コア部門（コアのみ）／総合部門（コア+オプション） |
| ブラケットUX | **ハイブリッド**：モバイル=ラウンド送りステッパー／デスクトップ=フルブラケット（R32拡張） |
| ナビ | **「予想」タブをハブ化**。コア3行＋オプション入口カード。読み取り専用「グループ」タブは残す |
| 採点 | **シンプル版**（各カテゴリ1ルール、基本+1）。下記§5 |

## 3. ベスト32トーナメント構造（確定データ）

`X1`=グループX1位（=`groupRank[X][0]`）, `X2`=2位（=`groupRank[X][1]`）, `wc(...)`=指定グループ群の3位ワイルドカード枠。

| 試合 | 上シード | 下シード |
|---|---|---|
| M1  | E1 | **wc(A/B/C/D/F)** |
| M2  | I1 | **wc(C/D/F/G/H)** |
| M3  | A2 | B2 |
| M4  | F1 | C2 |
| M5  | K2 | L2 |
| M6  | H1 | J2 |
| M7  | D1 | **wc(B/E/F/I/J)** |
| M8  | G1 | **wc(A/E/H/I/J)** |
| M9  | C1 | F2 |
| M10 | E2 | I2 |
| M11 | A1 | **wc(C/E/F/H/I)** |
| M12 | L1 | **wc(E/H/I/J/K)** |
| M13 | J1 | H2 |
| M14 | D2 | G2 |
| M15 | B1 | **wc(E/F/G/I/J)** |
| M16 | K1 | **wc(D/E/I/J/L)** |

ワイルドカード枠は8つ：**M1, M2, M7, M8, M11, M12, M15, M16**。

R16へのペアリング：`M1×M2, M3×M4, M5×M6, M7×M8, M9×M10, M11×M12, M13×M14, M15×M16` の勝者同士。以降 QF→SF→決勝。

この構造は不変の定数として `public/data.js` に `BRACKET_STRUCTURE`（各R32試合の `[topSeed, bottomSeed]`、bottomSeed はグループコード or `{wildcard: ['A','B',...]}`）として持つ。

## 4. データモデル

### 4.1 予想（メンバー毎 `pred`）

既存フィールドは維持し、オプション3フィールドを追加する。

```js
{
  // ---- コア（現状維持・独立）----
  champion: null,        // 優勝（チームコード）
  runnerUp: null,        // 準優勝
  topScorer: '',         // 得点王（自由入力）

  // ---- オプション（Phase B 追加。未着手なら空）----
  // ① グループ順位：各組の [1位, 2位, 3位]（順序が意味を持つ。4位は GROUPS[K] の残り1チーム）
  groupRank: { A: [], B: [], /* … */ L: [] },

  // ② 3位ワイルドカード割当：枠ID → 3位チームコード
  thirdAssign: { M1: null, M2: null, M7: null, M8: null,
                 M11: null, M12: null, M15: null, M16: null },

  // ③ ノックアウト勝ち上がり：各ラウンドの勝者
  knockout: {
    r32: [],  // 16要素：M1..M16 の勝者（=16強到達チーム）
    r16: [],  // 8要素：R16の勝者（=8強到達）
    qf:  [],  // 4要素：QFの勝者（=4強到達）
    sf:  [],  // 2要素：SFの勝者（=決勝進出＝2人の決勝進出者）
  },
}
```

- **旧 `bracket`（r16/qf/sf/final）は廃止**。旧 R16 ベースの `BracketScreen` / `R16_TEAMS` / `deriveBracket` は撤去または `knockout` 用に作り直す。SEEDの旧 `bracket` データは無視/削除。
- 決勝の勝者＝優勝、敗者＝準優勝に相当するが、**champion/runnerUp はコアの独立ピック**。ノックアウト予想は SF 勝者（＝2人の決勝進出者）までで完結し、決勝の勝敗予想はコアに委ねる。ブラケット画面の決勝列は「コアの優勝予想」を読み取り表示する締め（任意）。

### 4.2 対戦カードの導出（`deriveKnockout`）

純関数で `groupRank` + `thirdAssign` → 全ラウンドの対戦カードと、整合済みの勝者配列を返す（既存 `deriveBracket` と同じ「上流変更で下流の無効勝者を消す」方針）。

1. R32の各試合のカードを `BRACKET_STRUCTURE` から組む：
   - 上/下シードが `X1`/`X2` → `groupRank[X][0]` / `groupRank[X][1]`
   - ワイルドカード下シード → `thirdAssign[slot]`
2. `knockout.r32[i]` が当該カードの2チームに含まれなければ無効化（null）。
3. R16カード = R32ペアの勝者同士。以降同様に QF/SF を整合。

未確定（groupRank が埋まっていない等）のスロットは「未定」プレースホルダ表示。

### 4.3 3位割当の制約

- `thirdAssign[slot]` は、その枠の許可グループ群のいずれかの**3位チーム**（`groupRank[g][2]`）のみ選択可。
- 同一チームを複数枠に割り当て不可（使用済みは選択肢から無効化）。
- 割当に使える3位が未予想（`groupRank[g][2]` が空）のグループは候補に出ない。

### 4.4 永続化・マイグレーション

- `emptyPred()` に上記オプション3フィールドの空初期値を追加。
- `load()`：既存 localStorage / KV の `pred` にオプションフィールドが無ければ空で補完（後方互換）。旧 `bracket` フィールドは読み飛ばす。
- KV共有設定（`fetchConfig`）側は予想データを持たない（予想はクライアント localStorage 管理の現行方式を踏襲）。

## 5. 採点（シンプル確定版）

各カテゴリ「1ルール」、基本「ピタリ当たり1つ＝+1」。

| 部門 | カテゴリ | ルール | 配点 | 満点 |
|---|---|---|---|---|
| コア | 優勝 | 的中 | +25 | |
| コア | 準優勝 | 的中 | +15 | |
| コア | 得点王 | 的中（trim一致） | +20 | コア計 **60** |
| オプション | グループ順位 | 各組で順位がピタリのチーム1つにつき +1（1位/2位/3位が対象） | +1 | 3×12 = **36** |
| オプション | ノックアウト | 勝ち上がり的中チーム1つにつき +1（各ラウンド到達：16強/8強/4強/決勝進出） | +1 | 16+8+4+2 = **30** |

- **順位採点**：`groupRank[K][i] === GROUP_RESULT[K][i]`（i=0,1,2）で +1。GROUP_RESULT は admin入力の実順位（Phase A 実装済み、先頭から1位）。
- **ノックアウト採点**：各ラウンド `r32/r16/qf/sf` について、`knockout[R]` の各チームが「実際にそのラウンドへ勝ち上がったか」を集合判定し +1。深く勝ち残るチームは各ラウンドで積み上がり自然に高得点（決勝進出を当てると 16強+8強+4強+決勝進出 で実質+4）。
- 決勝の勝者（優勝）はコアで採点するため二重計上しない。**3位決定戦は対象外**。
- ノックアウトの実結果は、採点用に actual 側（`RESULT` 相当）へ `r32/r16/qf/sf` の到達チーム集合が必要。**admin入力UIは後続タスク**（本specのUI実装時はサンプル結果で動作確認）。

### スコア構造（`scoreMember` の返り値拡張）

```js
{
  core: { champion, runnerUp, topScorer, total },        // 最大60
  option: { groupRank, knockout, total },                 // 最大66
  coreTotal,                  // = core.total
  grandTotal,                 // = core.total + option.total（総合部門）
  // 内訳（バッジ表示用）：順位の的中組数、各ラウンドのヒット数 等
}
```

## 6. UI設計

### 6.1 「予想」タブ＝ハブ（`InputScreen` 拡張）
- 上部：コア3行（優勝/準優勝/得点王。現状のタップ→シート選択を維持）。
- 下部「オプション予想」セクション：
  - 「グループ順位予想」カード（進捗 `n/12`）→ §6.2 へ
  - 「3位ワイルドカード」カード（進捗 `n/8`。順位予想で3位が揃うほど割当可能に）→ §6.3 へ
  - 「ノックアウト予想」カード（順位予想・3位割当が未完なら「順位予想後」表示でロック）→ §6.4 へ
- タブ数はモバイル4／デスクトップ5のまま据え置き。オプション3画面はハブからのサブ画面（戻る導線付き）。

### 6.2 グループ順位予想（順番タップ式・新規 `screens-grouprank.jsx`）
- 12組をカードで縦に。各カード：4チームを並べ、タップ順に「1位🥇/2位🥈/3位」バッジ付与。再タップで取消、後続順位が繰り上がる。3位まで決まれば4位は自動（淡色表示）。
- 全体進捗 `n/12`。スワイプ/次へで組を流す導線。

### 6.3 3位ワイルドカード割当（新規 `screens-thirdwild.jsx`）
- 8枠（M1/M2/M7/M8/M11/M12/M15/M16）を縦リスト。各枠に「許可グループ群」ラベルと、その3位チームの選択チップ（使用済み・未予想は無効）。
- 8枠埋まれば完了。R32カードが確定し、ノックアウト予想が解放される。

### 6.4 ノックアウト予想（ハイブリッド・`screens-knockout.jsx`）
- 対戦カードは `deriveKnockout` で自動。ユーザーは勝者タップのみ。
- **モバイル**：ラウンド送りステッパー。1ラウンドの試合を縦リスト、勝者タップ→「次のラウンドへ」、進捗ドット。R32(16)→R16(8)→QF(4)→SF(2)。決勝は「コアの優勝予想」を読み取り表示。
- **デスクトップ**：フルブラケット（既存SVG盤面を R32 まで拡張、幅フィット）。
- 既存レスポンシブ基盤 `modeFor`/`useContainerWidth` を流用。共通の MatchCard / TeamRow 部品を両表示で共有。

### 6.5 ランキング（`RankingScreen` 拡張）
- 上部に2部門タブ：**コア部門**（`coreTotal` で集計・全員）／**総合部門**（`grandTotal` で集計）。
- 各行に内訳（コア / 順位 / ノックアウト）と、オプション参加者の `OPT` バッジ。
- 表彰台・展開詳細は既存パターンを踏襲し、選択部門のスコアで描画。

## 8. テスト方針

- 純関数の単体テスト（`functions/_lib` と同様に `*.test.js`）：
  - `deriveKnockout`：未確定スロットの扱い、上流変更時の下流勝者無効化、ワイルドカード反映。
  - `scoreMember`：コア／順位／ノックアウト各満点ケース、部分的中、オプション未着手で `grandTotal===coreTotal`。
  - 3位割当の制約（許可グループ外/重複/未予想3位）。
- 既存 `validate.test.js` のスタイルに合わせる。
- 80%+ カバレッジ目標（純ロジック中心）。
