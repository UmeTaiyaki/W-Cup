# グループ突破/敗退の数学的確定（クリンチ）表示 設計

- 日付: 2026-06-19
- 対象: 結果タブの順位表 + トーナメント表（ブラケット）
- ステータス: 設計（レビュー待ち）

## 1. 目的・背景

W杯2026はグループステージ進行中（開幕6/11）。メキシコのように**全試合は終わっていないが数学的にグループ突破が確定**しているチームがある。現状アプリには「全試合終了で順位確定」と「全チーム1試合以上消化後の暫定順位（緑線）」しかなく、進行中の数学的確定を示せない。

本機能で以下を実現する:

1. **結果タブの順位表**に、数学的に確定したチームの「確定バッジ」（1位確定 / 突破確定 / 敗退確定）を表示する。
2. **トーナメント表**に、順位が一意に確定したチームを該当スロット（A1 / A2 等）へ実チーム名として配置する。

## 2. 方針（確定した設計判断）

| 論点 | 決定 |
|------|------|
| 確定の判定方法 | **数学的クリンチ判定（自前計算）**。残り試合の全結果シナリオを列挙 |
| ブラケット反映の粒度 | **順位まで一意確定した枠のみ**実チーム名を配置（誤表示ゼロ優先） |
| 結果タブのバッジ | **突破確定（2位以内）** と **敗退確定**。加えて **1位確定**（ブラケットA1配置に必要なため内部判定し、バッジも表示） |
| 3位通過枠 | 全12グループの3位成績比較が必要で確定判定が極めて複雑なため **v1対象外**。自グループ内順位（1位/2位/敗退）のみ判定 |
| GD依存の確定 | v1は**勝点ベースの保守判定**（同点はタイブレーク負け側とみなす）。GDで初めて確定するケースは「未確定」扱い → **確定と表示したものは必ず正しい**を保証 |

## 3. クリンチ判定アルゴリズム

### 3.1 入力

- グループメンバー: `window.WC.GROUPS[g]`（4チームのコード配列。例 A=["MEX","KOR","RSA","CZE"]）
- 消化試合: `window.WC.GROUP_MATCHES[g]`（`[{a,b,ga,gb,status}]`）

### 3.2 残り試合の導出

- グループは4チーム総当たり=全6ペアリング（`generateFixtures(members)` で生成可能）。
- **確定済み（FT）** の試合のみを「決着済み」とみなす。
- 残り試合 = 全6ペアリング − FT済みペアリング。**LIVE は未確定（残り扱い）** とする（試合中の結果で確定を主張しない安全側）。

### 3.3 シナリオ列挙

- 残り試合 n 件それぞれに `W / D / L`（home視点）の3通りを割り当て、3^n 通りを列挙。n≤6 なので最大 729 通り（実際は数件で軽量）。
- 各シナリオで、FT済み結果 + 仮定した残り結果から **各チームの最終勝点** を算出する（勝点のみ。GDは使わない）。

### 3.4 判定ロジック（保守的・勝点ベース）

同点は「タイブレークで負ける」と仮定する。あるチーム X について、各シナリオで「X以上の勝点を持つ他チーム数」を `geCount`、「X超の勝点を持つ他チーム数」を `gtCount` とする。

- **突破確定（2位以内）**: すべてのシナリオで `geCount ≤ 1`
  （X以上が常に1チーム以下 → X は最悪でも2位 → 突破確定）
- **1位確定**: すべてのシナリオで `geCount = 0`
  （X以上の他チームが常にゼロ → X は常に単独1位）
- **2位で確定（secondLocked / ブラケットA2用）**: 突破確定 **かつ** すべてのシナリオで `gtCount ≥ 1`
  （常にXより上が最低1チーム → 1位になれない。突破確定と合わせて「ちょうど2位」確定）
- **敗退確定**: すべてのシナリオで `gtCount ≥ 2`
  （Xより上が常に2チーム以上 → 最良でも3位以下 → 2位以内不可能）

> 保守性の根拠: 「X以上(`≥`)」を上位候補に数えることで、同点のタイブレーク結果が不利でも安全。よって `geCount≤1` が全シナリオで成り立つなら、X は確実に top2。逆向きに `gtCount≥2` が全シナリオで成り立つなら、X は確実に top2 不可能。

- 純ロジック（副作用なし）。既存 `generateFixtures` を残り試合導出に再利用。
- FTのみ集計するため、`status` を見て FT 以外を残り扱いにする小ヘルパを内包。

### 3.6 ユニットテスト（純ロジックなので単体で検証可能）

代表ケース:
- 2勝＋他全敗で勝点上、残り1試合でも追いつけない → `qualified`/場合により`won`
- メキシコ実データ相当（A組 FT結果を流し込み）→ 期待ステータス
- 最終戦のみ残り、勝点同値で並ぶ可能性 → 保守的に `qualified` にしない
- 全試合FT → top2=qualified, 3位以下=eliminated（既存 computeStandings と整合）
- 残り試合0で勝点が一意 → 1位=won, 2位=secondLocked

## 4. 結果タブ（`public/screens-group.jsx` `LeagueTables`）

- 各グループの順位表描画前に `computeClinchStatus(members, matches)` を1回算出。
- 各行（チーム）にバッジを付与:
  - `won` → 🥇 **1位確定**（金系）
  - `won` でない `qualified` → ✅ **突破**（緑系）
  - `eliminated` → ⬇️ **敗退**（グレー系）
  - いずれでもない → バッジなし
- 既存の「暫定突破圏（緑の縦線）」はそのまま併用（暫定 vs 確定は別概念）。
- バッジは順位表の既存テーマ色トークンを使い、行レイアウトを崩さない小さなチップとして配置。

## 5. トーナメント表（`public/screens-knockout.jsx` / `public/lib/bracket.js`）

- 目的: 全グループ終了を待たず、**順位が一意確定したチーム**を実結果としてブラケットの該当スロットに表示。
- スロット解決:
  - `won` のチーム → そのグループの **1位スロット（例 A1）**
  - `secondLocked` のチーム → **2位スロット（例 A2）**
  - 上記いずれも満たさないスロット → 現状の表示（TBD/予想）を維持
- 3位ワイルドカード枠は全グループ比較が必要なため**従来どおり**（`RESULT.groupResult` 全確定後の既存ロジックに委ねる）。
- 既存のブラケット導出（`deriveKnockout` / `RESULT.groupResult`）と整合させるため、確定スロットから部分的な `groupRank` を組み立てて供給する。具体的な配線は実装計画で確定する（既存の予想表示・実結果オーバーレイの仕組みを壊さない形で、確定枠のみ上書き）。

## 6. データフロー

```
GROUPS (data.js) + GROUP_MATCHES (/api/results 経由)
        │
        ▼
computeAllClinch()  ──→  { g: { code: {qualified, won, eliminated, secondLocked} } }
        │                         │
        ▼                         ▼
LeagueTables（バッジ）      knockout（確定スロットへ配置）
```

- 追加のAPI・DB変更は不要。既存の `window.WC.GROUPS` / `window.WC.GROUP_MATCHES` のみで完結する。

## 7. エラー処理・エッジケース

- メンバー欠落/空グループ → 空ステータスを返す（バッジなし）。
- `GROUP_MATCHES` 未取得（watch無効/取得失敗）→ 全試合未消化と同じ＝バッジなし（graceful degradation）。
- スコアが片方欠落の試合 → 未消化扱い（既存 `isNum` 判定に倣う）。
- LIVE中に一時的に確定→非確定に揺れない（LIVEは常に残り扱いのため確定はFT進行に対して単調）。

## 8. スコープ外（YAGNI）

- 3位通過枠の数学的確定（全グループ横断比較）。
- GDで初めて確定するケースの確定表示（保守判定で未確定のまま）。
- 採点ロジックへの影響（本機能は表示のみ。`provisionalGroupResult` 等の採点系は変更しない）。
