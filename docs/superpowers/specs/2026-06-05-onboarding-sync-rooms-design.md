# オンボーディング・同期コード・部屋（ルーム）設計

- 日付: 2026-06-05
- 対象アプリ: W杯予想（Cloudflare Pages `wcup2026-yosou`、KV `CONFIG`）
- ステータス: 設計確定（実装計画へ）

## 1. 背景と目的

現状は単一の共有ドキュメント `predictions:v1`（`members[]` ＋ `preds{}`）に全員の予想が入り、アプリを開いた全員が同じボードを見る「仲間内アプリ」構造。端末をまたいだ「自分の予想」の継続ができず、ユーザー識別もない。

ログインの壁による離脱を避けることを最優先に、**Googleログインは採用せず**、匿名の「同期コード」方式でユーザー識別と端末またぎ同期を実現する。あわせて、予想を見比べるための「部屋（ルーム）」を導入する。

### 解決したい課題（優先順）
1. ログインの壁で離脱されないこと（最優先）
2. 端末をまたいで自分の予想を継続できること
3. 仲間で予想を見比べられること（部屋）

## 2. スコープ

### 今回実装する
- オンボーディング・ウィザード（名前 → コア予想 → オプション予想(スキップ可) → 完了）
- ユーザー identity ＋ 同期コード（発行・表示・別端末復元）
- 部屋: 作成 / 参加 / 見比べビュー
- 上記を支える KV データモデルと API、純ロジックのテスト

### 今回の対象外（YAGNI）
- 部屋からの退出・部屋の削除
- 本格的なレート制限基盤
- 本人認証（メールマジックリンク等）・Googleログイン
- 予想を部屋ごとに変える運用（予想は1ユーザー1つ・全部屋共有）

## 3. 用語とコンセプト

- **User（ユーザー）**: アプリを使う1人。名前・予想1つ・同期コードを持つ。
- **同期コード（user code）**: 「この端末＝自分」を別端末で復元するための秘密のコード。bearer 相当（知っている人＝本人扱い）。URL には載せない。
- **Room（部屋）**: 予想を見比べるための集まり。参加コードを持ち、複数の User が所属する。
- **参加コード（room code）**: 部屋に参加するためのコード。共有してよい（招待用）。
- 予想は **1ユーザー1つ**。複数の部屋に同じ予想を持ち込んで見比べる。
- 今までの「全員共有の1ボード」は「**部屋の中の見比べビュー**」に相当する。

## 4. ユーザー体験 / フロー

### 4.1 正面玄関（ルーティング）
アプリ起動時に `localStorage` の `{userId, code}` を判定する。
- **あり** → ホーム（自分の予想 ＋ 所属部屋一覧 ＋ 作る/参加 導線）へ直行。オンボーディングはスキップ。
- **なし**（初回 or 別ブラウザ/キャッシュ削除）→ オンボーディング開始。

### 4.2 オンボーディング・ウィザード
1. **名前を決める**: ニックネーム入力。確定時にサーバで `user` を作成し同期コードを発行、`localStorage` に `{userId, code}` を保存（壁なし・そのまま次へ）。
2. **コア予想**: 優勝 → 準優勝 → 得点王。
3. **オプション予想**: グループ順位 → サードプレイス → トーナメント。**スキップして完了に進める**。
4. **完了画面**:
   - 完了メッセージ
   - 同期コード表示（コピー＋QR、「**このブラウザを変えると消えます**／無くすと復元できません」の注意）
   - 「部屋を作る」「部屋に参加する」導線

### 4.3 別端末での同期
新しい端末/ブラウザで「別の端末から続ける」→ 同期コードを入力 → `usercode:{code}` から userId を解決 → user を取得して `localStorage` に保存 → 以降その端末は自分として扱う。

### 4.4 部屋
- **作る**: 部屋名を入力 → `room` を作成し参加コードを発行 → 自分を members に追加 → 参加コード/QR を表示して招待。
- **参加**: 参加コードを入力 → `roomcode:{code}` から roomId を解決 → 自分の userId を members に追加。
- **見比べビュー**: room を読み、members の各 user を取得して一覧表示（既存の複数メンバー比較 UI を流用）。

### 4.5 編集・プライバシー
- 各 User は **自分の予想だけ**編集できる。部屋では他人の予想は閲覧のみ。
- 同期コード/userId が軽い bearer 保護として機能する。厳密な認証は行わない（離脱回避を優先）。

## 5. データモデル（KV: 既存 `CONFIG` ネームスペースを流用）

新規キー4種を追加する。既存 `predictions:v1` はテストデータのため移行せず、新モデルで作り直す。

| キー | 値 | 用途 |
|------|----|------|
| `user:{userId}` | `User` JSON | ユーザー本体（名前・予想・コード） |
| `usercode:{CODE}` | `userId`（文字列） | 同期コードからの逆引き |
| `room:{roomId}` | `Room` JSON | 部屋本体（部屋名・members・コード） |
| `roomcode:{CODE}` | `roomId`（文字列） | 参加コードからの逆引き |

### 5.1 User
```
{
  version: 1,
  id: string,            // 内部ID（衝突しないランダム）
  name: string,          // ニックネーム
  code: string,          // 同期コード（正規化済み、ハイフンなしで保持）
  pred: Pred,            // 既存 predictions.js の予想構造（core + option）
  updatedAt: string      // ISO8601
}
```

### 5.2 Room
```
{
  version: 1,
  id: string,
  name: string,          // 部屋名
  code: string,          // 参加コード（正規化済み）
  members: string[],     // 所属 userId の配列
  ownerId: string,       // 作成者 userId
  createdAt: string,
  updatedAt: string
}
```

### 5.3 コード仕様
- 文字種: 紛らわしい文字（`0/O`、`1/I/L` 等）を除いた Crockford Base32 系。
- 長さ: 8文字（約40bit）。
- 表示: `XXXX-XXXX`（ハイフン区切り）。保持・比較はハイフン除去＋大文字化で正規化。
- 同期コードは秘密（URL/ログに載せない）。参加コードは共有可（招待用、QR/コピー）。
- 生成時に KV を確認して**衝突したら再生成**する。

## 6. API エンドポイント（Cloudflare Pages Functions）

既存の `_lib/http.js`（`json()` ヘルパ）と入口バリデーション（`content-length` 上限・`LIMITS`）を踏襲する。

### 6.1 `functions/api/user.js`
- `POST {op:'create', name}` → user 作成、`{userId, code, user}` を返す
- `POST {op:'setPred', userId, pred}` → 自分の予想を更新（`validatePred` で正規化）
- `POST {op:'sync', code}` → 同期コードから user を復元して返す
- `GET ?id={userId}` → user 取得（起動時ロード用）

### 6.2 `functions/api/room.js`
- `POST {op:'create', userId, name}` → room 作成、`{roomId, code, room}` を返す
- `POST {op:'join', userId, code}` → members に userId を追加（重複追加しない）、room を返す
- `GET ?id={roomId}` → room ＋ members 各 user を同梱（見比べボード用）

### 6.3 共通方針
- 認証なしのため、配列長・文字列長・payload サイズに上限を設けて KV 肥大化を防ぐ（既存 `LIMITS` を拡張）。
- 競合は last-write-wins ＋ `updatedAt` 表示。書き込み時は対象キーのみ更新。
- `sync` / `join` は推測対策として軽い保護を意識（コード長40bit を主防御とし、過剰なレート制限基盤は今回作らない）。

## 7. 純ロジックとテスト

既存の `node:test` ＋ `assert/strict` 流儀に合わせ、純関数に切り出してテストする。

- `functions/_lib/codes.js`: `generateCode()` / `formatCode()` / `normalizeCode()`。
  - テスト: 出力が許可文字種・長さ8、`formatCode` のハイフン区切り、`normalizeCode` の大文字化＋ハイフン/空白除去、紛らわしい文字を含まないこと。
- `functions/_lib/users.js`: User の生成・検証・正規化（`makeUser`, `validateUser` 等）。
- `functions/_lib/rooms.js`: Room の生成・検証・メンバー追加（`makeRoom`, `addMember`：重複防止・上限）。
- 予想の検証は既存 `predictions.js` の `validatePred` / `emptyPred` を再利用。

## 8. フロントエンド

既存の `public/*.jsx`（スクリプトタグ読み込みの React、`window.WC` 名前空間）と `screens-*.jsx` のパターンに合わせる。

- オンボーディング・ウィザード画面（名前 / コア / オプション / 完了）。
- 同期コード表示（コピー＋QR）。QR は軽量実装。難しければ v1 はコピーのみで開始し QR は後追い可。
- 別端末同期（コード入力）画面。
- 部屋: 作成 / 参加 / 見比べビュー（既存の比較 UI を流用）。
- ルーティング: `localStorage` の有無でオンボーディング/ホームを分岐。

## 9. 受け入れ基準

- 初回アクセスで名前 → コア → （オプションをスキップ可能に）→ 完了 までウィザードで進める。
- 完了画面に同期コードと「ブラウザを変えると消える」注意、部屋の作る/参加導線が出る。
- 別ブラウザで同期コードを入力すると、自分の予想が復元される。
- 部屋を作成して参加コードを発行でき、別ユーザーがそのコードで参加できる。
- 部屋ビューで所属メンバー全員の予想を見比べられ、自分の予想のみ編集できる。
- `codes.js` / `users.js` / `rooms.js` の純ロジックにテストがあり通る。

## 10. リスク・留意点

- 同期コードを無くすと復元不可（仕様。完了画面で明示）。
- bearer コードのため、コードを知る人はなりすませる（賞品等の厳密性が必要になれば別途認証を検討）。
- 部屋ビューはメンバー数ぶんの KV 読み取りが発生（上限内なら許容）。
- 認証なしのため上限値設定とサイズ制限を徹底する。
